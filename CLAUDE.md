# CLAUDE.md — 教材を「作る・直す」側への指示

このファイルは、**この研修教材リポジトリ自体を編集する人（教材メンテナ・講師・Claude などの作業エージェント）** 向けの指示です。
新人（受講者）に向けた指示や、研修中の伴走AIへの指示ではありません。それらは別ファイルです。役割を取り違えないでください。

| ファイル | 誰のためのものか | 役割 |
|---|---|---|
| **このファイル（CLAUDE.md）** | **教材を作る/直す人** | 教材を編集するときの設計思想・不変条件・整合性ルール |
| [AGENTS.md](AGENTS.md) | 研修中の伴走AI（Antigravity） | 新人に伴走するときのふるまいの憲法 |
| [training/README.md](training/README.md) | 新人（受講者） | 研修の進め方・AI利用ルール・質問の作法 |
| [training/for-instructors.md](training/for-instructors.md) | 講師 | 対面の進め方・評価観点・見守りどころ |

> つまり: **CLAUDE.md は「教材の作り手の視点」、AGENTS.md は「研修運用中のAIの視点」**。混ぜない。

---

## 0. この研修の根本思想（編集時に絶対に崩さないこと）

この教材は、次の一貫した設計の上に成り立っています。**どの編集も、この思想の延長線上に置いてください。**

1. **3ヶ月・2フェーズ構成**。前半（週0〜2）で「自分の手で書く力」を固め、週3以降（週3〜12）で「AIを正しく使う力」を育てる。**この順番が教材の背骨**です。
2. **配属先はAIを業務利用してよい現場**。だからゴールは「AIを使わない人」ではなく「**AIの出力を見極め、既存システムに安全に参画できる人**」。
3. **AIが出したものを評価・修正できるのは、自分で書ける人だけ**。だから前半はあえてAIを禁じる。この因果を弱める編集をしてはいけない。
4. **自習中心・リモート前提**。すべての課題は新人が**一人で正解判定できる**こと（テストが通る／期待出力と一致する／指定レスポンスが返る）。新設課題にも必ず自己採点手段を付ける。
5. **技術力と同じ重さで「動き方」を育てる**。報連相・原因の切り分け・レビュー往復は、技術と同等以上に重視する。AI時代でも人間に残る仕事だから。

## 1. 絶対に守る不変条件（ここを壊すと教材が破綻する）

教材を編集するとき、次の整合性を**必ず**保ってください。1つの変更が複数ファイルに波及します。

### ① フェーズ境界の一貫性（最重要）
**AIにコードを書かせてよい境界は「週2と週3のあいだ」** です。週0〜2＝禁止（手書き徹底）、週3〜12＝解禁（AI協働）。
- 第1フェーズ（手書き）= 週0〜2、第2フェーズ（AI協働）= 週3〜12。基礎の手書きは「言語の素振り」が終わる週2まで、週3（Spring Boot）からAIを解禁し、現場のAI活用を長く練習させる設計。
- この境界に触れる変更をするなら、**最低でも次の全部を同時に直す**:
  - [training/curriculum.md](training/curriculum.md)（全体像・各週のAI利用区分）
  - [training/README.md](training/README.md)（「2. AI利用ルール」のフェーズ別記述）
  - [AGENTS.md](AGENTS.md)（§3 フェーズ別のコード生成ルール）
  - [training/for-instructors.md](training/for-instructors.md)（フェーズごとの見守りどころ・評価観点）
  - 該当する週の `weekX-.../README.md`（冒頭の「AI利用区分」）
  - [training/PROGRESS.md](training/PROGRESS.md)（フェーズの区切り表示）
- **どれか1つだけ直して他を放置するのは厳禁**。新人やAI伴走者が矛盾した指示を受ける。

### ② 週の順序と依存
週は「前の週の上に積み上がる」。順序は
`week0-setup → week00-it-basics → week1-java-basics → week2-sql-db → week3-springboot → week4-test-and-final → week5-ai-collab → week6-code-reading → week7-safe-modification → week8-test-quality-ai → week9-11-project → week12-final`。
- 週を追加・削除・並べ替えるなら、curriculum.md のスケジュール表・README のフォルダ順・PROGRESS.md のチェックリストを揃える。

### ③ 自己採点可能性
新しい課題（exercise）を足すなら、必ず:
- 「検証方法」（叩けば正誤が分かるコマンド／期待出力／テスト）を書く。
- 対応する `solutions/` に模範解答を置く（新人は先に見ない前提だが、講師と自己確認のため必要）。
- 「つまずきポイントとヒント」を添える（答えではなくヒント）。

### ④ 用語のかみくだき
新人はIT未経験。新しい専門用語を出したら、その場で**用語メモ**（たとえ話つき）を添える。既存の `> 用語メモ:` の書式に合わせる。

### ⑤ PROGRESS.md と AGENTS.md の連動
AGENTS.md は「伴走AIが PROGRESS.md を source of truth として読む」前提で書かれている。
- PROGRESS.md の構造（現在地3行・完了チェックリスト）を変えるなら、AGENTS.md の §0・§2 の参照と齟齬が出ないか確認する。

## 2. ファイルの地図（どこに何があるか）

```
リポジトリ直下/
  CLAUDE.md            ← このファイル（教材を作る側への指示）
  AGENTS.md            ← 伴走AIへの指示（新人に接するときの憲法）
  training/
    README.md          ← 新人向けトップ（進め方・AI利用ルール・質問テンプレ・日報）
    curriculum.md      ← 3ヶ月の全体像と各週の到達目標
    for-instructors.md ← 講師向け（対面の進め方・評価観点）
    PROGRESS.md        ← 新人の進捗状態（伴走AIが読み書きする source of truth）
    week0-setup/ … week12-final/  ← 各週（README + exercises/ + solutions/）
```

各週フォルダは原則 `README.md`（狙いと課題一覧）/ `exercises/`（課題）/ `solutions/`（模範解答）の3点構成。

## 3. 第2フェーズ（週5〜12）の教材を作るときの指針

第2フェーズはまだ整備途中の部分があります。新規に作るときは次を守ってください。

- **題材リポを1本用意し、週6〜11で使い回す**。週6（読解）→週7（改修）→週8（テスト）→週9〜11（参画）が**同じ既存リポ**を対象にすることで、「既存システムに参画する」体験が連続する。バラバラの題材にしない。
- 各課題に「**AI生成物のレビュー観点を当てる欄／検証ノート**」を設ける（curriculum.md の「AI生成コードのレビュー観点」を流用）。コードを書かせて終わり、にさせない作り。
- 週9〜11は**PRとレビュー往復が主役**。「一発で完成」ではなく、講師（レビュアー役）が指摘を返す前提の課題設計にする。
- ハルシネーション・リグレッション・影響範囲といった「AIを使う上での落とし穴」を、座学ではなく**手を動かして踏ませる**課題にする。

## 4. 文体・トーン（教材を書くときの作法）

- 新人向けドキュメント（README系・週README・課題）は、**やさしく・具体的に・脅さない**。IT未経験が読んで安心できる温度。
- 抽象論で終わらせない。「こうしてください」だけでなく「こう叩けば確認できます」まで書く。
- 既存の書式（見出し記法・`> 用語メモ:`・表・チェックリスト）に揃える。**この教材に既にある美学を、自分の都合で崩さない**。
- 安易な近道（とりあえず1ファイルだけ直す等）を選ぶときは、§1の不変条件に照らし、波及先を明示してから決める。黙って楽な方へ流れない。

---

## 5. Git 運用

コミットメッセージは [プレフィックス]: の形式（`feat:` `fix:` `refactor:` `docs:` `chore:` など。教材なので `docs:` と `feat:` が中心）。本文に変更点を箇条書き。
- 例: `feat: 研修を3ヶ月・2フェーズ構成に拡張しAI協働フェーズを追加`
- フェーズ境界や不変条件に関わる変更は、コミット本文に「波及して直したファイル」を列挙すると後から追える。
