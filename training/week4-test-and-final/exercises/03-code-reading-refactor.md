# 課題3: コードリーディング + 小改修（他人のコードを読んで直す）

**所要時間の目安**: 120〜180分

**AI利用区分**: ✅ AIにコードを書かせてよい。ただし**「どこが・なぜ悪いか」を、AIに頼る前にまず自分で言語化すること**。改修後コードは自分の言葉で説明できること。

---

## AI生成コードの説明記入欄（提出時に必ず埋める）

```
【自分で見つけた「悪いところ」と、その理由（最低4つ）】
1.
2.
3.
4.

【どう直したか（要点）】


【振る舞いを変えていないと言える根拠（どのテストで保証したか）】

```

---

## 1. 背景・狙い（現場でどう使うか）

現場であなたが触るコードの**ほとんどは「他人が書いた既存コード」**です。新規にゼロから書くより、「すでにあるコードを読んで、少し直す（改修する）」仕事のほうが圧倒的に多い。

このとき一番怖いのは、**直したつもりが、別の動きを壊してしまう**ことです。それを防ぐのが「振る舞いを固定するテスト」です。

> 用語メモ: **リファクタリング** … 外から見た動き（振る舞い）を変えずに、中身のコードを読みやすく・直しやすく整えること。
> 用語メモ: **characterization test（キャラクタライゼーションテスト / 仕様化テスト）** … 「今このコードが実際どう動くか」をそのまま写し取って固定したテスト。改修の前後で動きが変わっていないことの“安全網”になる。

---

## 2. 題材コード（`legacy-code/` にある。動くが、改善の余地がある）

`legacy-code/src/main/java/com/example/legacy/ShippingFeeCalculator.java`

このクラスは「配送料を計算する」もの。入力は `"地域,重量(g)"` という文字列のリスト。

- `calc(List<String> lines)` … 全行の配送料を合算する。
- `calcWithDiscount(List<String> lines, int memberRank)` … 合算後、会員ランクで割引する。

**このコードは一応“動きます”**（`legacy-code` で `./gradlew test` を実行すると、付属の characterization test が全部通る）。でも、現場のレビューなら必ず指摘が入る作りになっています。**どこが・なぜ悪いか、自分で探してください。**

> ヒント（答えではない）: 「同じことを2回書いていないか」「変数名から意味が読めるか」「エラーが起きたとき何が起きるか」「数字がそのまま埋め込まれていないか」あたりを疑って読むと見つかります。

---

## 3. やること（詳細手順）

### (a) 読む & 言語化する（先にこれをやる。AIに聞く前に！）
コードを読んで、「どこが・なぜ悪いか」を**最低4つ**、上の記入欄に自分の言葉で書く。ここを飛ばしてAIに「直して」と言うと、何も身につきません。

### (b) 振る舞いを変えずに改善する
`ShippingFeeCalculator.java` を改修する。改修の鉄則:

- **外から見た計算結果（戻り値）は1mmも変えない。**
- 付属の characterization test（`ShippingFeeCalculatorCharacterizationTest`）が、改修後も**全部緑のまま**であること。
- そのうえで、自分が(a)で挙げた問題を直す。

> 改修してよい例: メソッド/変数のリネーム、重複の共通化、数字の定数化、空のcatchをやめる（ただし「壊れた行はスキップして合算に含めない」という結果は保つこと。ここを変えるとテストが赤くなる）。

---

## 4. 完成条件（自分で判定できる）

- [ ] 「悪いところ」を最低4つ、理由つきで言語化した
- [ ] コードを改修した
- [ ] `legacy-code` で `./gradlew test` が**改修後も全部緑**（=振る舞いを壊していない）
- [ ] 重複コードが解消された／変数名が意味のあるものになった／マジックナンバーが定数化された／空のcatchがなくなった、のうち**少なくとも3つ**が直っている
- [ ] 説明記入欄を埋めた

---

## 5. 検証方法

`legacy-code`（`build.gradle` がある階層）で:

```bash
./gradlew test
```

**期待結果**: 改修の前も後も `BUILD SUCCESSFUL`。

> 大事な確認の仕方: **改修する前に一度** `./gradlew test` を流して「緑であること」を見ておく。改修しながら何度も流し、**ずっと緑のまま**を保つ。途中で赤くなったら、それは振る舞いを変えてしまったサインなので、すぐ戻す。これが現場のリファクタリングのやり方そのものです。

---

## 6. つまずきポイントとヒント

<details>
<summary>「重複コード」がどこか分からない</summary>

`calc` と `calcWithDiscount` の中をよく見比べてください。地域と重量から料金を出す部分が、ほぼそのままコピペされています。これを1つのメソッド（例: `feeForLine` や `feeOf`）に切り出せば、1か所直すだけで両方に効くようになります。
</details>

<details>
<summary>空のcatchをどう直すか</summary>

`catch (Exception e) {}` は、エラーが起きても何も言わずに握りつぶしています。これだと「壊れた行があっても誰も気づけない」。改修では、**「壊れた行は合算に含めない」という結果は保ったまま**（=テストを赤くしない）、せめて原因をログ（標準エラー出力など）に残すようにします。`catch (Exception e)` のような広すぎる捕まえ方を、起きうる例外（数値変換失敗・要素不足）に絞るのも改善点です。
</details>

<details>
<summary>マジックナンバーって？</summary>

`500` `800` `2000` のように、コードに直接書かれた「意味の説明がない数字」のこと。`LIGHT_WEIGHT_LIMIT` のような定数名を付けると、何の数字か読み手に伝わります。
</details>

<details>
<summary>改修したらテストが赤くなった</summary>

振る舞いを変えてしまっています。よくあるのは「壊れた行をスキップする挙動」を変えてしまうケース。テストの失敗メッセージ（期待値 vs 実際値）を読んで、どのケースで結果が変わったか特定し、元の結果に戻してください。
</details>
