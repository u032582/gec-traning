# 3ヶ月カリキュラム — 全体像と到達目標

このファイルは、研修全体の地図です。「いま自分がどこにいて、次に何ができるようになるのか」を確認するために使ってください。
進め方・AI利用ルール・質問の仕方は [README.md](README.md) を見てください。

> **この研修が前提とする現場**: あなたは **AIエージェント（Claude / Copilot / Cursor など）を業務で使ってよい現場** に配属されます。だからこのカリキュラムのゴールは「AIを使わずに全部手で書ける人」ではありません。**AIを正しく使いこなして、既存のシステムに参画し、安全に直して、レビューを通して提出できる人** です。
> ただし——それでも前半は「自分の手で書く力」を徹底的に鍛えます。**AIが出したものを評価・修正できるのは、自分が書ける人だけ**だからです。順番が命です。

---

## 全体の流れ（3ヶ月 = 約13週）

この研修は **2つのフェーズ** でできています。

```
━━━━━━━━━━ 第1フェーズ: 基礎を「自分の手で」固める（週0〜2） ━━━━━━━━━━
週0前半 環境構築      : 道具をそろえる（PCで開発できる状態にする）＋clone・MPE表示
週0後半 IT基礎(座学)  : ITの「言葉」と「仕組み」をイメージできるようになる
週1     Java基礎     : 自分の手でJavaが書けるようになる
週2     SQLとDBアクセス: データベースからデータを出し入れできるようになる

━━━━━━━━━━ 第2フェーズ: AIを「正しく」使って現場に立つ（週3〜12） ━━━━━━━━━━
週3     Spring Boot  : Webの裏側（API）を1本作れるようになる（ここからAI解禁）
週4     テスト＋総合課題: テストを書き、実装〜提出まで一人で回せるようになる
週5     AI協働の作法  : AIに任せる/任せないの線引き、プロンプト、生成物の検証
週6     既存コード読解 : 他人が書いた大きめのコードを、AIを相棒に読み解く
週7     安全な改修    : 影響範囲を見極め、既存リポを壊さずに直す（バグ修正・小機能追加）
週8     テストと品質をAIで: AIにテストを書かせ、人間が品質を担保する運用
週9-11  実戦プロジェクト: 既存リポへの参画を模した中規模課題（複数機能・PR・レビュー往復）
週12    総仕上げ・評価 : 現場想定の総合演習と振り返り
```

- **第1フェーズ（週0〜2・基礎）**: ここはAIにコードを書かせません。自分の手で書けるようになることが、第2フェーズでAIを使いこなす土台になります。手書きは「言語の素振り」が終わる週2まで。
- **第2フェーズ（週3〜12・AI協働）**: 週3（Spring Boot）からAIを解禁します。ただし「貼るだけ」ではなく、**読む・検証する・直す・説明する**を徹底します。週3〜4で「AI活用の素振り」をし、週5以降で「AI協働の作法・既存リポへの参画」を深めていきます。

各週は「前の週の上に積み上がる」設計です。順番を飛ばさないでください。
**いちばん最初は週0前半の「環境構築」**です。この研修ドキュメントはCursorのMPE拡張で表示して読む想定なので、先に環境を整え、リポジトリを `git clone` する必要があります（手順はSETUP.md末尾）。

## このカリキュラムが目指す最終到達点

3ヶ月後、あなたが次の状態になっていることを目指します。

> **「AIを相棒に、既存のシステム（自分が書いたのではない、大きめのコードベース）に参画し、仕様を読み、影響範囲を見極めて安全に改修・小機能追加を行い、テストを書き、PRを出し、レビューの往復をこなして取り込まれるまでを、一人で回せる」**

これは現場に配属された新人が、最初の数ヶ月で実際に任される動きそのものです。「ゼロから一人で全部作る」ことは求めません。**既存の流れに乗って、壊さず、確実に前に進める**ことを目指します。

## 各週の到達目標（この状態になればクリア）

### 【第1フェーズ：基礎を自分の手で固める（週0〜2）】

#### 週0後半: IT基礎（座学）
- **到達目標**: ITの基本の言葉（ターミナル/シェル/プロセス/サーバー/パス等）を**自分の言葉で説明でき**、開発の全体像をイメージできる。
- **完了の判定**: [week00-it-basics/README.md](week00-it-basics/README.md) の各章の理解チェックとハンズオン小課題をクリア。

#### 週1: Java基礎を実装で固める

> **ここまでが第1フェーズ（手書き徹底）**です。週2の修了が、第2フェーズに進む前提。基礎の素振り（言語とSQL）を自分の手で終えてから、週3でAIを解禁します。手が動かないままAIを解禁すると、「中身が分からないまま貼るだけの人」になり、かえって伸びません。

### 【第2フェーズ：AIを正しく使って現場に立つ（週3〜12）】

> このフェーズからAIエージェント（Agent）モードと補完を解禁します。ただし **「生成物を読む・検証する・説明する」が絶対条件** です。詳細は [README.md](README.md) の「AI利用ルール（第2フェーズ）」。
> 週3〜4は、まだ自分が手を動かす比重も高い「**AI活用の素振り**」です。週5以降で、AI協働の作法と既存リポへの参画を本格的に深めます。

#### 週3: Spring BootでREST API（ここからAI解禁）
- **到達目標**: 「詳細設計を渡されたら、APIを1本実装できる」。
- **完了の判定**: 指定のリクエストを送ると、設計どおりのレスポンスが返る。
- **AI利用区分**: ✅ 解禁。ただし生成コードを自分の言葉で説明できること。

#### 週4: 単体テスト＋総合課題＋コードリーディング
- **到達目標**: 「現場の実装工程を一通り自走できる」（実装〜テスト〜PR提出）。
- **完了の判定**: 総合課題のAPIが設計どおり動き、テストが通り、PRが出せている。
- **AI利用区分**: ✅ 解禁。ただし生成コードを説明でき、コードリーディング課題では「どこが・なぜ悪いか」を自分で言語化してから直すこと。

#### 週5: AI協働の作法（AIに任せる/任せないの線引き）
- **学ぶこと**: AIエージェントの基本操作、プロンプトの書き方（仕様・制約・期待出力を渡す）、生成コードの**レビューと検証**、AIが間違える典型パターン（それっぽいが動かない／古い書き方／ハルシネーション）、「任せてよい作業／自分で判断すべき作業」の線引き。
- **到達目標**: 「AIにコードを書かせ、それを**自分の言葉で説明でき、間違いを見つけて直せる**」。
- **やり方**: 既習の週1〜4の課題を、今度は**AIに書かせて→検証して→説明する**形でやり直す。手で書いた自分の答え（週1〜2）と比較し、AIの癖を体で知る。
- **完了の判定**: AI生成コードのレビュー観点（後述）を埋めた「検証ノート」を提出できる。
- **AI利用区分**: ✅ 解禁。ただし検証ノート必須。

> 用語メモ: **ハルシネーション** … AIが、それっぽいが事実でない（存在しないメソッド名・古いAPIなど）を自信満々に出力すること。AIを使う以上、これを見抜く力が必須。

#### 週6: 既存コードを読み解く（AIを相棒に）
- **学ぶこと**: 大きめのコードベース（自分が書いたのではないSpring Bootアプリ）の歩き方、エントリポイントから処理を追う、3層構造のどこに何があるかを地図化する、AIに「このコードは何をしているか」を要約させてから**自分で裏取りする**読み方。
- **到達目標**: 「初見の中規模リポを渡されたら、AIの助けも借りて**全体像と該当箇所を自力で把握できる**」。
- **やり方**: 題材リポ（後述の実戦プロジェクトのベース）を渡す。仕様書なしで「この機能はどこで実装されているか」を追う探索課題を複数こなす。AIの要約を鵜呑みにせず、必ずコードで確認する。
- **完了の判定**: 指定された機能の処理の流れ（Controller→Service→Repository→SQL）を、ファイル名・行を挙げて説明できる。
- **AI利用区分**: ✅ 読解の相棒として活用。ただし**最終的な理解は自分の言葉で**。

#### 週7: 安全な改修（壊さずに直す）
- **学ぶこと**: バグ修正と小機能追加、変更前に**影響範囲（どこに波及するか）を見極める**、既存のコード規約・命名に合わせる、変更を小さく保つ、変更前後でテストを通す（リグレッション確認）、AIに改修案を出させて**安全性を自分で判断する**。
- **到達目標**: 「既存リポのバグ票・小さな機能要望を渡されたら、**周りを壊さずに**直して、テストを通せる」。
- **やり方**: 週6の題材リポに対し、バグ修正課題と小機能追加課題を複数。各課題に「これが直ればOK」のテストが付いている。AIに頼ってよいが、**なぜその変更で直るのか／他に影響が出ないか**を説明できること。
- **完了の判定**: 改修後、対象テストが通り、かつ既存テストが壊れていない（全テストグリーン）。
- **AI利用区分**: ✅ 解禁。ただし影響範囲の説明と全テストグリーンが条件。

> 用語メモ: **リグレッション** … 直したつもりが、別の動いていた箇所を壊してしまうこと。既存テストを流して、壊していないか確認するのが基本動作。**影響範囲** … その変更が波及する範囲。「この関数を直すと、呼んでいる側はどこか」を追うこと。

> **Gitの進め方（週7以降ずっと共通）**: この研修では、現場（SES＝客先の社内リポにメンバーとして参画する形）に合わせて、**同じリポをcloneし、作業用のブランチを切ってPRを出す**やり方で統一します。具体的には `ブランチを切る → fetch origin で origin/master を最新化 → 実装 → push → PR → レビュー往復 → master へマージ`。
> OSSでよく使う「フォーク（fork）」運用は、**SES現場ではほぼ使わない**ので、この研修では扱いません。`fork` や `upstream` という言葉は今は覚えなくて大丈夫です。研修リポのmasterはPR必須に保護してあるので、自然とこの流れになります。

#### 週8: テストと品質をAIで担保する
- **学ぶこと**: AIに単体テストを書かせる／不足ケースを指摘させる、テストの**良し悪しを人間が判断する**（カバレッジの数字に騙されない、意味のあるアサーション）、AIにコードレビューさせて指摘の妥当性を選別する、静かな失敗（silent failure）・例外握りつぶしを見抜く。
- **到達目標**: 「AIにテストを書かせ・レビューさせつつ、**最終的な品質判断は自分が持てる**」。
- **やり方**: 週7で改修したコードに、AIと協働でテストを足す。AIの出したテスト・レビュー指摘を「採用/却下」し、その理由を書く。
- **完了の判定**: AI生成テストに対する取捨選択の記録（採用理由・却下理由）を提出でき、テストが意味のある検証になっている。
- **AI利用区分**: ✅ 解禁。ただし品質判断は人間が持つ。

#### 週9〜11: 実戦プロジェクト（既存リポへの参画を模した中規模課題）
- **学ぶこと**: ここまでの全部を統合。既存リポに対し、**複数の機能追加・改修をPR単位で進める**。設計の意図を汲む、コミットを意味のある単位で切る、PR本文で変更を説明する、**レビュー指摘に往復で対応する**（修正→再レビュー）。
- **到達目標**: 「**既存システムへの参画を一人で回せる**——仕様を読み、AIを使って実装・テストし、PRを出し、レビューの往復をこなして取り込まれるところまで」。
- **やり方**: 3週間で、題材リポに対し複数のタスク（機能追加・バグ修正・改善）をこなす。各タスクはPRで提出し、講師がレビュアー役として**実際に指摘を返す**。指摘に対応して再提出する往復を必ず経験する。
- **完了の判定**: 各PRがレビューを経てマージ可能な状態になる。レビュー往復の記録が残っている。
- **AI利用区分**: ✅ 全面解禁（現場想定）。説明責任は常に伴う。

#### 週12: 総仕上げ・評価
- **学ぶこと**: 現場想定の総合演習（短い仕様だけ渡されて、調査〜実装〜テスト〜PRまで一人で）。3ヶ月の振り返り。
- **到達目標**: 最終到達点（上記）に達したかを、実演で確認する。
- **完了の判定**: 総合演習を、報連相とレビュー対応も含めて自走でやり切れる。

## AI生成コードのレビュー観点（第2フェーズで毎回使う）

第2フェーズでは、AIが書いたコードをそのまま信じません。次の観点で**毎回チェック**してください。これは現場でAIを使う人の「安全装置」です。

| 観点 | 問い |
|---|---|
| **動くか** | 本当にビルドが通り、テストが通るか？（AIは動かないコードも自信満々に出す） |
| **正しいか** | 仕様どおりか？ 異常系・境界値を取りこぼしていないか？ |
| **存在するか** | 使われているメソッド・APIは実在するか？（ハルシネーション） |
| **古くないか** | 非推奨・時代遅れの書き方ではないか？ |
| **馴染むか** | 既存コードの規約・命名・構造に合っているか？（浮いていないか） |
| **壊さないか** | 既存の動いている箇所に影響しないか？（リグレッション） |
| **説明できるか** | なぜこの実装なのか、自分の言葉で言えるか？ |

> 1つでも「No」または「分からない」があれば、それは**まだ提出してはいけない**サインです。
