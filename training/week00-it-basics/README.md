# 週0後半: IT基礎 — 環境構築で触れた「言葉」と「仕組み」を理解する

> このフォルダは、**環境構築（[week0-setup/SETUP.md](../week0-setup/SETUP.md)）を終えたあと**に読む座学（ざがく＝読んで理解する学習）です。まだ環境構築が済んでいない人は、先にそちらを終わらせてください。

## なぜこれをやるのか

環境構築では、「ターミナル」を開き、「コマンド」を打ち、「ファイルパス」を指定し、PostgreSQLを「起動」しました。手順どおりに進めて動いたはずですが、**「自分がいま何をしたのか」までは、まだぼんやりしている**のではないでしょうか。それで正常です。

この章は、その**ぼんやりを解消する「復習」**です。「ターミナル」「サーバー」「コンパイル」といった言葉が、それぞれ何を指し、裏でどんな仕組みが動いているのかを、腰を据えて理解します。**「環境構築で打った、あのコマンドはこういう意味だったのか」「あの黒い画面はこういうものだったのか」**とつながっていく感覚を大事にしてください。

ここで言葉と仕組みの土台を固めておくと、この先の週1〜4で出てくる用語が「ああ、あれのことか」とスムーズに入ります。**急がば回れ**です。1〜2日かけてじっくり読んでください。

> 用語メモ: **座学** … 手を動かす作業の合間に、本や資料を読んで知識・考え方を学ぶこと。この章は読み物が中心ですが、最後に手を動かす小課題もあります。

## 第1章: コンピュータは「ハード」と「ソフト」でできている

コンピュータ（PC）は、大きく2つに分かれます。

- **ハードウェア（ハード）** … 手で触れる物理的な部品。例: 画面、キーボード、本体の中の部品。
  - **CPU（シーピーユー）** … 計算を行う「頭脳」。プログラムの命令を実際に実行する場所。
  - **メモリ（RAM）** … 作業中のデータを一時的に置く「作業机」。電源を切ると消える。広い机ほど一度にたくさん作業できる。
  - **ストレージ（SSD / HDD）** … データを保存しておく「引き出し・倉庫」。電源を切っても消えない。ファイルはここに保存される。
- **ソフトウェア（ソフト）** … 手で触れない、命令やデータのかたまり。例: アプリ、OS、あなたが書くプログラム。

> たとえ話: ハードは「料理人とキッチン（人・道具）」、ソフトは「レシピ（手順書）」。レシピ（ソフト）があっても、料理人とキッチン（ハード）が無ければ料理はできません。逆も同じです。

**メモリとストレージの違いは、後で必ず効いてくる**ので覚えておいてください。
- メモリ = 机の上（速いが、電源を切ると消える、容量は小さめ）
- ストレージ = 引き出し（電源を切っても残る、容量は大きい、机より遅い）

## 第2章: OS（オーエス）— コンピュータの「土台ソフト」

> 用語メモ: **OS（オペレーティングシステム）** … コンピュータ全体を動かす一番土台のソフト。ハードとアプリの間に立って、両者を仲介する。

あなたが普段使っている **Windows** はOSの一種です。ほかに **macOS**（Macのパソコン）、**Linux（リナックス）**、スマホの **Android / iOS** などがあります。

OSの役割は、たとえるなら「**会社の総務部**」です。アプリ（社員）が「ファイルを保存したい」「インターネットに繋ぎたい」とお願いすると、OSがハード（設備）を手配してくれます。アプリはハードを直接いじらず、必ずOSを通します。

### この研修で重要な事実: 現場のサーバーはほぼ Linux
あなたが作るプログラムは、最終的に **サーバー**（後述）の上で動きます。そして**現場のサーバーのOSは、ほとんどが Linux** です。Windowsしか使ったことがない人がほとんどなので、この研修では **WSL2** という仕組みを使って、Windowsの中でLinuxを動かして練習します（詳しくは環境構築の章で）。

- **Windows** … あなたの手元のPCのOS。普段の操作に使う。
- **Linux** … 現場のサーバーのOS。プログラムを動かす本番の場所。**今のうちに慣れておくと現場で強い。**

> 用語メモ: **WSL2** … Windowsの中でLinuxを動かす仕組み。これを使うと、Windows PC1台で「Windowsの操作」と「Linuxでの開発」の両方ができる。

## 第3章: プロセス — 「動いているプログラム」

**プログラム**とは、コンピュータへの命令を書いたファイルのことです（くわしくは第6章で扱いますが、ここではこの理解で十分です）。では、そのプログラムが**実際に動いている状態**を何と呼ぶのでしょうか。それが **プロセス** です。

> 用語メモ: **プロセス** … OSの上で**実際に動いている1つのプログラム**のこと。プログラムが「紙に書かれたレシピ」なら、プロセスは「いままさに作っている料理」。

### 「プログラム」と「プロセス」は別物
この2つの違いが、プロセスを理解する第一歩です。

- **プログラム** … ストレージ（引き出し）に保存された、命令の書かれたファイル。**まだ動いていない**。レシピそのもの。
- **プロセス** … そのプログラムをOSが**メモリ（机）に読み込んで実行している状態**。動いている。料理中の状態。

> たとえ話: `Hello.java` のようなプログラムのファイルを、OSが動かし始めると（その具体的な手順＝コンパイルや実行は第6章で学びます）、この「動いている1回ぶん」が1つのプロセスになります。同じプログラムを2回起動すれば、プロセスは2つになります（同じレシピで2皿同時に作るイメージ）。

### OSは「たくさんのプロセスを同時に世話している」
あなたのPCでは、今この瞬間も、ブラウザ・音楽アプリ・OS自身の管理機能…と、**何十・何百ものプロセスが同時に動いています**。OS（第2章の「総務部」）の大事な仕事の1つが、この**プロセスたちにCPUとメモリを順番に割り当てて、同時に動かす**ことです。

- 各プロセスには、OSが付ける **PID（ピーアイディー＝プロセスID）** という番号があります。「いま動いているプロセスの整理番号」です。後でプロセスを止めたり調べたりするときに使います。

> 用語メモ: **PID** … 動いているプロセス1つ1つに付く識別番号（プロセスID）。同じプログラムでも、起動するたびに違うPIDが付く。

### なぜこれを今学ぶのか — 後の章にずっと効く
「プロセス」が分かっていると、後で出てくる次の話がスッと入ります。

- **サーバーは「ずっと動き続けるプロセス」**（第7章）。リクエストを待ち受けるプログラムを起動しっぱなしにしたもの＝サーバープロセス。
- **データベースもプロセス**。環境構築で出てくる `sudo service postgresql start` は、「PostgreSQLというプログラムを**プロセスとして起動する**」という意味です。起動していない＝プロセスが動いていない＝接続できない（`Connection refused`）。
- **WSL2を再起動するとPostgreSQLが止まる**のは、再起動でプロセスがいったん消えるから。だから毎回 `start` で起動し直す必要があります。
- あなたが週3で作る **バックエンドのプログラムも、起動するとプロセスになって**、リクエストを待ち受けます。

> 💡 つまり「何かを start する／起動する」という操作は、ほぼ「**プログラムをプロセスとして動かし始める**」という意味だと思ってOKです。

- [ ] OSがプロセスに対して何をしているか（CPU/メモリの割り当て、同時実行）を言える

## 第4章: ファイルとフォルダ、そして「パス」

データは **ファイル** という単位で保存されます。ファイルは **フォルダ**（ディレクトリとも呼ぶ）にまとめて整理します。フォルダの中にフォルダを作ることもでき、これが入れ子（階層）になっています。

> 用語メモ: **ディレクトリ** … フォルダのこと。ITの世界、とくにLinuxやプログラミングではフォルダをよくこの名前で呼ぶ。**フォルダ＝ディレクトリ**と覚えてOK。

### 拡張子（かくちょうし）— ファイルの「種類を表す名札」
ファイル名の最後にある `.txt` や `.java` の部分を **拡張子** と呼びます。これでファイルの種類が分かります。
- `memo.txt` … テキスト（ただの文章）ファイル
- `Hello.java` … Javaのプログラムが書かれたファイル
- `photo.jpg` … 画像ファイル

> あなたが書くJavaのプログラムは `.java` という拡張子のファイルに保存します。

### パス — 「ファイルの住所」
**パス**とは、ファイルやフォルダが「どこにあるか」を表す住所のような文字列です。これはプログラミングで**毎日使う最重要概念**です。

- **絶対パス（ぜったいパス）** … 一番上から全部書いた住所。
  - Linux/WSL2の例: `/home/shinjin/hello/Hello.java`
  - （都道府県から書いた住所と同じ。どこから見ても場所が一意に決まる）
- **相対パス（そうたいパス）** … 「いま自分がいる場所」から見た住所。
  - 例: いま `/home/shinjin/` にいるなら、`hello/Hello.java` で同じファイルを指せる
  - （「ここから2軒先」と言うのと同じ。基準が変わると指す先も変わる）

Linuxのパスの記号:
- `/` … フォルダの区切り（Windowsは `\` だが、Linux/WSL2では `/`）
- `.`（ドット1つ）… 「いまいるフォルダ（カレントディレクトリ）」
- `..`（ドット2つ）… 「1つ上のフォルダ」
- `~`（チルダ）… 「自分のホームフォルダ」（例: `/home/shinjin`）

> 用語メモ: **カレントディレクトリ** … 「いま自分が作業している場所のフォルダ」。ターミナル（次章）では、常に「今どこのフォルダにいるか」を意識します。

- [ ] 絶対パスと相対パスの違いを、住所のたとえで説明できる
- [ ] `.` `..` `~` `/` がそれぞれ何を意味するか言える

## 第5章: ターミナルとコマンド — 「文字でPCに命令する」

普段あなたは、マウスでアイコンをクリックしてPCを操作しています。これを **GUI（ジーユーアイ）** と言います。一方、**文字を打ち込んで命令する**操作方法を **CUI（シーユーアイ）／CLI（シーエルアイ）** と言い、その入力画面を **ターミナル** と呼びます。

> 用語メモ: **GUI** … アイコンやボタンを見て、マウスで操作する方式（Graphical User Interface）。普段のPC操作。
> 用語メモ: **CUI / CLI** … 文字でコマンドを打って操作する方式（Character/Command Line Interface）。**ターミナル**（黒い画面）で行う。
> 用語メモ: **ターミナル** … コマンドを打ち込む画面。「黒い画面」と呼ばれることも。WSL2では Ubuntu のターミナルを使う。

### なぜ「黒い画面」で操作するのか
最初は不安に感じますが、開発の現場ではターミナルが必須です。理由は:
- **正確で速い**（同じ操作を何度でも、間違いなく繰り返せる）
- **サーバーにはGUIが無いことが多い**（Linuxサーバーは文字で操作するのが普通）
- **手順を記録・共有できる**（コマンドをコピペで渡せる）

つまり「黒い画面が使える」こと自体が、現場の入口の必須スキルです。

### ターミナルと「シェル」は別物

ここで初心者がよく混同するポイントを、先に整理しておきます。**ターミナル（画面）** と **シェル（命令を解釈する係）** は別物です。

- **ターミナル** … コマンドを打ち込み、結果を表示する **「窓口（画面）」** そのもの。
- **シェル** … あなたが打った文字を受け取り、**「これはこういう命令だな」と解釈して、OSに実行させる係**。ターミナルの裏側で働いています。

> 用語メモ: **シェル（shell）** … ターミナルに打たれたコマンドを解釈し、OSに実行を依頼して、結果を返してくれるプログラム。直訳すると「殻（から）」で、利用者とOSの間をつなぐ外側の殻、というイメージ。

> たとえ話: **ターミナルは「受付カウンター（窓口）」**、**シェルは「そのカウンターに座っている受付係」**です。あなた（利用者）がカウンターで「`ls` を見せて」とお願いすると、受付係（シェル）がその意味を理解して、奥（OS）に取りに行き、結果をカウンターに並べてくれます。画面だけあっても、解釈してくれる係がいなければ何も起きません。

#### シェルにはいくつか種類がある（この研修は bash）
受付係（シェル）にもいくつか種類があり、少しずつ作法（文法）が違います。代表的なもの:
- **bash（バッシュ）** … Linuxで最も広く使われてきた定番のシェル。**この研修（WSL2 / Ubuntu）では bash を使います。**
- **zsh（ゼットエスエイチ）** … macOSの標準。bashと近いが少し高機能。
- **PowerShell（パワーシェル）** … Windows標準のシェル。文法がかなり違う。

> 💡 この研修の手順書で「**シェルは bash**」「**bashのターミナルで実行**」と書いてあるのは、この受付係として bash を使う、という意味です。ネットで調べ物をするときも、「bash コマンド」などと付けて検索すると、自分の環境に合った答えにたどり着きやすくなります（PowerShell向けの答えをそのまま打つと動かないことがあるため）。

> 用語メモ: **シェルスクリプト** … よく使うコマンドの並びを1つのファイルにまとめて、まとめて実行できるようにしたもの。「受付係への作業手順書」。研修で出てくる `gradlew`（ビルドを動かすスクリプト）もこの仲間です。

### コマンドの形
ターミナルに打ち込む命令を **コマンド** と呼びます。多くはこんな形です。
```
コマンド名  オプション  対象
```
例（よく使う基本コマンド。環境構築でもう何度か打ったはずです）:
- `pwd` … いま自分がどのフォルダにいるか表示する（カレントディレクトリを表示）
- `ls` … いまのフォルダの中にあるファイル・フォルダ一覧を表示する
- `cd フォルダ名` … 指定のフォルダに移動する（cd = change directory）
- `mkdir フォルダ名` … 新しいフォルダを作る（make directory）

> 💡 行頭の `$` はコマンドの例によく付きますが、`$` 自体は打ちません。その後ろを打ちます。

### 記号の読み方 — 「声に出して言える」ようにしておく
ターミナルや手順書には、ふだん名前を意識しない記号がたくさん出てきます。**記号の読み方が分からないと、AIに質問したり、ネットで調べたりするときに言葉にできず詰まります**。よく出るものを先に押さえておきましょう（無理に暗記しなくてOK。「ここに戻れば分かる」で十分）。

| 記号 | 読み方 | この研修での意味・出てくる場面 |
|---|---|---|
| `~` | チルダ | 自分のホームフォルダ（第4章）。`cd ~` でホームに戻る |
| `/` | スラッシュ | フォルダの区切り（第4章）。Linuxのパスで毎回使う |
| `\` | バックスラッシュ／円記号 | Windowsのパス区切り。Linux/WSL2では基本使わない |
| `.` | ドット／ピリオド | いまいるフォルダ（第4章）。拡張子の前にも付く（`Hello.java`） |
| `..` | ドットドット | 1つ上のフォルダ（第4章）。`cd ..` で戻る |
| `$` | ダラー／ドル | コマンド例の行頭の目印（打たない）。あとで「環境変数」でも使う |
| `#` | シャープ／ハッシュ | コマンド例の中の**コメント**（メモ書き。実行されない） |
| `-` | ハイフン／マイナス | コマンドの**オプション**に付く（例: `ls -l`） |
| `*` | アスタリスク／star | 「何でもいい」を表す（例: `*.java` で「.javaで終わる全部」） |
| `\|` | パイプ／縦棒（バーティカルバー） | コマンドの結果を次のコマンドに渡す（週が進むと出てくる） |
| `>` | 大なり／リダイレクト | 結果をファイルに書き出す。手順書では「次に進む矢印」の意味でも使う |
| `&` | アンド／アンパサンド | 「ついでに〜も」の意味で使われる記号 |
| `" "` | ダブルクォート | 文字列（文字のかたまり）を囲む。Javaでも使う（週1） |
| `' '` | シングルクォート | 同じく文字を囲む。用途で使い分ける |

> 💡 ネットやAIで質問するとき、`~` を「チルダ」、`|` を「パイプ」と**名前で言える**だけで、ぐっと調べやすくなります。記号をそのまま読み上げられない人は意外と多いので、ここで差がつきます。

## 第6章: プログラムが「動く」とはどういうことか

あなたがこれから書くのは **プログラム** です。プログラムとは「コンピュータへの命令を、順番に書いたもの」です。料理のレシピや、家具の組み立て説明書に似ています。

> 用語メモ: **プログラム** … コンピュータにやってほしい処理を、決まった文法（プログラミング言語）で書いた命令の集まり。
> 用語メモ: **プログラミング言語** … プログラムを書くための言葉。この研修では **Java（ジャバ）** を使う。

### 人間の言葉 → コンピュータの言葉への「翻訳」
コンピュータが直接理解できるのは、最終的には `0` と `1` の世界（機械語）だけです。人間が書いた `Hello.java` のようなプログラムは、そのままでは動きません。間に「翻訳」の工程が入ります。Javaの場合は2段階です。

1. **コンパイル** … 人間が書いた `.java` を、コンピュータ寄りの形（`.class`）に変換する。担当: `javac` というコマンド。
2. **実行** … 変換した `.class` を実際に動かす。担当: `java` というコマンド。

> 用語メモ: **コンパイル** … 人が書いたコードを、コンピュータが実行できる形に変換する作業。**ビルド**とほぼ同じ意味で使われることも多い。
> 用語メモ: **エラー** … プログラムの間違いや異常。コンパイル時に出る（文法ミス）ものと、実行時に出る（動かしている最中の問題）ものがある。**エラーは敵ではなく、コンピュータからの「ここが変だよ」というメッセージ**。読めるようになることが上達の近道。

### バグとデバッグ
- **バグ** … プログラムの中の間違い・不具合。
- **デバッグ** … バグを見つけて直す作業。

プログラマの仕事は「コードを書く」よりも「**動かない原因を調べて直す**」時間の方が長いくらいです。だからこの研修では「エラーを読む・原因を切り分ける」力を重視します。

## 第7章: ネットワークとサーバー — 「裏側」で働くプログラム

あなたが配属後に作るのは **バックエンド（裏側）** のプログラムです。これを理解するには、ネットワークとサーバーの仕組みのイメージが要ります。

> 用語メモ: **ネットワーク** … 複数のコンピュータがつながって、データをやり取りできる状態。インターネットは世界中をつないだ巨大なネットワーク。

### クライアントとサーバー
- **クライアント** … サービスを「お願いする側」。例: あなたのスマホのアプリ、ブラウザ。
- **サーバー** … お願いに「応える側」。例: データを返したり保存したりする、常に動いているコンピュータ（とその上のプログラム）。

> たとえ話: レストランで、**お客さん（クライアント）** が「ハンバーグください」と注文すると、**店（サーバー）** が作って出します。クライアントは「注文（リクエスト）」を送り、サーバーは「料理（レスポンス）」を返します。

- **リクエスト** … クライアントからサーバーへの「お願い」。
- **レスポンス** … サーバーからクライアントへの「返事」。

### フロントエンドとバックエンド
- **フロントエンド** … 利用者の目に見える部分（画面、ボタン、見た目）。
- **バックエンド** … 目に見えない裏側（データの計算・保存・取り出しなど）。**あなたが担当するのはこちら。**

> この研修でフロントエンド（HTML/CSS等の見た目）を扱わないのは、あなたの配属先での担当が**バックエンド**だからです。裏側のプログラムが、注文を受けて、データベースとやり取りして、結果を返す——その流れを作れるようになるのが目標です。

### データベース
- **データベース（DB）** … データを整理して保存し、必要なときに取り出せる「専用の保管庫」。バックエンドはこのDBと頻繁にやり取りする。この研修では **PostgreSQL** というデータベースを使う（週2以降）。

> 全体の流れ（イメージ）:
> **クライアント**（注文）→ **バックエンド**（あなたが作るプログラム。判断・処理）→ **データベース**（データの出し入れ）→ **バックエンド**（結果をまとめる）→ **クライアント**（返事）

この絵が、週3で作る「REST API」のまさに正体です。今はぼんやりで構いません。

## ハンズオン総仕上げ（手を動かして確かめる）

第4章（パス）と第5章（コマンド）で学んだことを、実際に手を動かして確かめます。環境構築は済んでいるので、すぐに試せます。**WSL2のターミナル（Ubuntu）** を開いて、上から順に実行してください。

### 小課題: フォルダを作って移動し、現在地を確認する
```bash
# 1. いま自分がどこにいるか確認する
pwd

# 2. ホームに練習用フォルダ practice を作る
mkdir ~/practice

# 3. practice の中にさらに day1 フォルダを作る
mkdir ~/practice/day1

# 4. day1 に移動する
cd ~/practice/day1

# 5. もう一度、いまどこにいるか確認する
pwd

# 6. 1つ上のフォルダに戻る
cd ..

# 7. いまのフォルダの中身（day1 があるはず）を一覧表示する
ls
```

### 完成条件（自分で判定できる）
- 手順5の `pwd` が **`/home/（あなたのユーザー名）/practice/day1`** と表示される
- 手順7の `ls` の結果に **`day1`** が含まれている

この2つが確認できたら、第4章・第5章の内容が身についています。

### つまずきポイント
<details>
<summary>手順5のpwdが day1 になっていない</summary>

手順4の `cd ~/practice/day1` が正しく実行できていない可能性があります。タイプミス（スペルや `~` `/` の間違い）がないか確認して、手順2〜4をもう一度実行してください。それでもダメなら、エラーメッセージ全文をAIのAskモードに貼って「このエラーの意味を教えて」と聞いてみましょう（家庭教師用途なので許可されています）。
</details>

<details>
<summary>「mkdir: cannot create directory ... File exists」と出る</summary>

すでに同じ名前のフォルダがある、という意味のエラーです。練習をやり直している場合に出ます。問題ないので、そのまま次の手順に進んで構いません（または `rm -r ~/practice` でいったん消してからやり直してもOK。`rm -r` はフォルダごと削除するコマンドなので、消す対象を間違えないよう注意）。
</details>
